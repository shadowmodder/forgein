Why Forgein

The problem with
every Claude session.

Claude Code is powerful. But each session starts fresh — no knowledge of your workflow, your team norms, or the security patterns you care about. You re-explain. You re-install. You re-catch the same bugs.

01 — MEMORY
Claude's memory is write-only

Claude Code builds context files over time but gives you no way to read, search, or manage them. Sessions diverge. Stale context accumulates. Forgein mem is the only CRUD interface for your cross-session memory.

02 — SKILLS
Generic skill stores don't know your workflow

Other catalogs give everyone the same recommendations. Forgein reads your CLAUDE.md, git log, and memory files — then matches against those signals. Your last 20 commits are better signal than any survey.

03 — SECURITY
Security checks are a separate tool you forget to run

Forgein sec runs inside your Claude session — it's a natural step before you push, not a context switch. Eight vulnerability classes checked against staged changes, ranked by severity, with one-line attack scenarios.

Three commands

Each one solves
a real session failure.

These aren't wrappers. Each command solves a gap that Claude Code doesn't address: skills that drift, memory that's opaque, security that gets skipped.

/forgein optimize

Reads your workflow.
Recommends what you'll actually use.

Fetches the registry, inventories your installed skills, then reads your CLAUDE.md + memory + git log as one context string. Scores each skill by whole-word signal matching — not fuzzy, not substring. "pr" won't match "sprint."

  • Word-boundary scoring: char before/after each match must be non-alphanumeric
  • Already-installed skills excluded automatically
  • Default Enter = install all — no hunting for the right input
  • Fetches via gh API — no CDN lag, works offline from GitHub

/forgein mem

Full control of
Claude's memory system.

List, search, add, prune, and audit your cross-session memory files. Prune verifies that referenced PRs, files, and functions still exist before you discard anything. Audit catches structural issues the auto-memory system never surfaces.

  • Groups by type: user · feedback · project · reference
  • Prune checks live artifacts — PRs, branches, file paths, functions
  • Audit finds broken index links, duplicates, missing frontmatter
  • Works across all project memory directories simultaneously

/forgein sec

Security check
built into your session.

Scans staged git changes — or any path you give it — against eight vulnerability classes. Ranks by severity. Shows a one-line attack scenario for every finding so you understand the actual risk, not just a flag name. Honest that it's a vibe check, not a full audit.

  • Secrets, command injection, SQL injection, path traversal, SSRF
  • XSS, insecure deserialization, hardcoded credentials
  • No staged changes? Asks for a path — never scans silently
  • Critical → high → medium severity ranking with attack scenarios

How optimize works

Signal-matching, not guessing.

optimize doesn't ask what you need. It reads evidence of what you do — then scores every registry skill against that evidence with word-boundary precision.

01 — FETCH
Registry
Pulls registry.json via gh api. Falls back to raw GitHub. Parses the skills array.
02 — READ
Your context
CLAUDE.md (global + project), memory index files, git log --oneline -20. One string.
03 — SCORE
Word-boundary match
For each skill: count whole-word matches of its signals + tags. Char before/after must be non-alphanumeric.
04 — INSTALL
Top 5 → one step
Ranked by score. Enter to install all. gh api fetch + write to ~/.claude/commands/.
# score(skill) = Σ whole_word_match(term, context) for term in signals + tags # # whole_word: char at position[i-1] and [i+len] must be non-alphanumeric (or boundary) # "pr" → matches "open prs" ✓ "pr" → does NOT match "sprint" ✗ # "git" → matches "git log" ✓ "git" → does NOT match "digital" ✗ # context = read(~/.claude/CLAUDE.md) + read(./CLAUDE.md) + find(~/.claude/projects/**/MEMORY.md) + git log --oneline -20
